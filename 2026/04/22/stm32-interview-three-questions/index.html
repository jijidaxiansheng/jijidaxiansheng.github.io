面试复盘：STM32 底层结构和 Flash 读写，我是怎么重新梳理的

发表于2026-04-22|编程基础
|浏览量:

最近面试时被问到了两个很典型的底层问题。一个偏芯片整体结构，一个偏存储器行为。表面上看，它们分别在问“STM32 里面到底有什么”和“Flash 为什么这么麻烦”，但本质上都在考一件事：你有没有把 STM32 当成一个真正的硬件系统来理解，而不只是会调库函数。

这篇文章就把这两个问题重新整理成一条更清晰的主线，也当作一次自己的复盘。

面试里的 2 个问题

这次遇到的问题大致是下面这 2 个方向：

  1. STM32 最基础的构成是什么，底层到底有哪些东西？
  2. Flash 在底层是怎么读写的？

如果单独回答，很容易变成零散知识点。但把它们连起来看，其实是在从“芯片整体结构”一步步往“存储器行为”深入。

一、先回答最根本的问题：STM32 底层到底由什么组成

很多时候我们写代码，只能看到 main()、外设初始化函数和中断回调，但这些代码真正跑起来，背后依赖的是 STM32 内部完整的硬件结构。

从最基础的角度看，STM32 可以拆成下面几部分：

  • 内核：也就是 Cortex-M 处理器，负责执行指令。
  • 存储器：包括 Flash 和 SRAM。
  • 总线：负责连接内核、存储器和外设。
  • 外设：GPIO、USART、TIM、ADC、SPI、I2C 这些模块都挂在总线上。

如果再说得直白一点：

  • 内核负责“算”
  • 存储器负责“放”
  • 总线负责“搬”
  • 外设负责“和外界交互”

这也是为什么底层开发的本质常常被概括成一句话：通过读写寄存器控制硬件行为。

因为不管是点亮一个 LED，还是发一个串口字节，本质上都是 CPU 经过总线去访问某个外设的寄存器。

二、Flash 为什么“好读不好写”

第二个问题通常会问得很直接：Flash 怎么读，怎么写？

1. Flash 为什么读起来很简单

在 STM32 里，Flash 是统一编址的，也就是 memory mapped。
这意味着 CPU 访问 Flash，并不需要一套特别复杂的“读取命令”，它更像是在访问一片普通的内存地址。

所以从“读”的角度看，Flash 的特点是：

  • 地址是固定可见的
  • CPU 可以直接按地址访问
  • 读操作通常很自然

2. Flash 为什么写起来麻烦

Flash 的难点不在读，而在写。

因为它不是像 SRAM 那样能随便改某一个字节。Flash 有自己的物理约束：

  • 写之前通常要先擦除
  • 擦除不是按一个字节擦，而是按页或扇区擦
  • 写入时通常也有对齐要求
  • 为了防止误操作，还要先解锁，写完后再上锁

所以底层写 Flash 的典型过程一般是：

  1. 解锁 Flash 控制器
  2. 擦除目标页或扇区
  3. 按规则写入数据
  4. 检查状态
  5. 上锁

如果面试官继续追问“那你用标准库怎么写一个最简单的 Flash 读写例子”，可以用下面这种思路回答：

这个例子里：

  • FLASH_Unlock()：先解锁 Flash 控制器
  • FLASH_ErasePage(addr)：先擦除目标页
  • FLASH_ProgramWord(addr, data)：按字写入数据
  • FLASH_Lock()：写完后重新上锁
  • *(__IO uint32_t *)addr：按地址直接把写进去的数据再读回来

如果想把这个例子讲得更像面试回答，可以再补一句：Flash 的读取像普通内存访问，真正麻烦的是写之前要先擦除，而且擦除粒度通常比写入粒度更大。

这也是面试里经常会继续追问的点：为什么 Flash 不能像变量那样随手赋值？

答案就在于它和 SRAM 的物理实现不一样。

3. 这个问题真正考什么

我觉得这个问题真正考的不是你会不会背 FLASH_Unlock() 这种函数名，而是你是否理解：

  • Flash 是非易失性存储
  • 它的写入有硬件限制
  • 擦除粒度和写入粒度并不一样
  • 底层写存储器必须考虑安全性和状态机

如果只会背库函数流程，很容易一换平台就不会了。

三、为什么这两个问题总是一起出现

复盘下来我发现，这两个问题虽然看起来分散，但其实很适合连着问。

因为它们刚好对应了嵌入式底层里两个很核心的理解层次：

  1. 先理解芯片整体结构
    也就是 CPU、总线、存储器、外设是怎么连起来的。

  2. 再理解存储器行为
    知道 Flash 和 SRAM 有什么差别，为什么一个容易读但不好写，一个适合当运行时内存。

如果只会写初始化代码，但不知道代码背后到底在访问谁、存储器为什么会受限，面试里通常很快就会被继续追问。

四、如果面试官继续追问，可以怎么往下答

如果这两个问题答完，对方通常还会继续往下延伸。比较常见的是：

  • STM32 的存储器映射大致怎么分布
  • SRAM 和 Flash 的用途分别是什么
  • 为什么程序能直接从 Flash 里运行
  • Flash 擦除为什么按页或扇区
  • 写 Flash 时如何避免覆盖已有数据

这些追问其实都在验证同一件事：
你到底是在“背 API”，还是已经能把 API 对应回硬件行为。

总结

这次面试里的这两个问题让我重新意识到，STM32 底层知识不能只靠零碎记忆。

更好的整理方式应该是：

  • 先从芯片结构入手
  • 再理解存储器特性
  • 再把 Flash 的读写逻辑放回硬件限制里看

这样你回答的时候就不会只停留在“库函数怎么调”，而是能讲清楚：

  • STM32 底层到底由哪些部分组成
  • 为什么 Flash 可以像内存一样直接读
  • 为什么 Flash 写起来一定要解锁、擦除、再写入
  • 为什么底层开发本质上是在和寄存器、总线和存储器打交道

对我来说，这两个问题不只是面试题，更像是一次提醒：
真正的底层能力，不是会几个函数，而是能把代码背后的硬件逻辑讲清楚。
