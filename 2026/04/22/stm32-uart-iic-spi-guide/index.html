STM32 通信协议梳理：UART、IIC、SPI 的构成、时序和收发例子

发表于2026-04-22|编程基础
|浏览量:

面试里一旦聊到 STM32 的外设，通信协议几乎绕不过去。最常见的就是 UART、IIC 和 SPI。很多时候大家会背初始化流程，也知道几个函数名，但一旦被追问“它们到底怎么传数据”“时序为什么这样设计”，就容易越说越乱。

所以我把这部分单独整理成一篇。目标不是把协议手册一股脑背下来，而是先把三种通信方式最核心的构成、时序、速率和收发过程讲清楚。

一、通信协议到底在“传”什么

如果先把问题压缩一下，通信协议本质上是在解决一件事：

硬件到底是怎么把内存里的 0 和 1 送到引脚上的？

答案的核心，是移位寄存器和时序规则。

1. 发送的本质

CPU 不会一位一位手动去翻转引脚。
通常是 CPU 先把一个字节写入外设的数据寄存器，然后：

  • 外设把数据装入移位寄存器
  • 再根据协议要求的时序
  • 一位一位地从引脚送出去

接收时则反过来：

  • 外设按照时序在引脚采样
  • 把采样到的位拼成字节
  • 放进接收寄存器
  • 再通过标志位或者中断通知 CPU 来取

这就是为什么很多通信外设都带有：

  • 数据寄存器
  • 状态寄存器
  • 中断标志
  • 波特率或时钟配置寄存器

因为这些寄存器就是硬件通信过程的“控制面板”。

二、先把 UART、IIC、SPI 的基本构成捋清楚

如果面试里一上来就聊波形，很容易越说越乱。更稳的说法是先把三种常见串行通信的基本构成拆开。

UART

UART 是异步串口，最核心的构成只有两类：

  • TX：发送线
  • RX：接收线

如果是最常见的全双工场景，那就是双方各自一根发、一根收。它没有独立时钟线，所以双方必须先约定好波特率、数据位、校验位和停止位。

可以把它理解成：靠预先约定的节奏，加上起始位和停止位，完成一帧一帧的数据传输。

IIC（I2C）

IIC 其实就是 I2C，面试里这两个叫法经常混用。它的基本构成是：

  • SCL：时钟线
  • SDA：数据线

它是同步通信，总线上的设备跟着 SCL 的节奏采样或输出数据。除此之外，IIC 还有几个很典型的协议元素：

  • 主机和从机
  • 7 位或 10 位地址
  • 应答位 ACK/NACK
  • 开始条件和停止条件

它更像是一条“共享总线”，不是谁想发就直接发，而是主机先发起，再按地址找到目标设备。

SPI

SPI 也是同步通信，但它和 IIC 的风格不一样。它的典型构成是：

  • SCLK：时钟线
  • MOSI：主机输出、从机输入
  • MISO：主机输入、从机输出
  • CS/SS：片选线

它的特点是结构直白、速率高、通常支持全双工。主机通过片选线决定当前在和哪个从机通信，然后跟着时钟把数据一位一位移出去、移进来。

可以把三者先记成一句话：

  • UART：没有时钟，靠约定波特率和帧格式
  • IIC：两根线共享总线，靠地址和应答工作
  • SPI：多一根片选线，跟着时钟高速移位

UART 时序图

UART 空闲时一般是高电平，发送开始先拉低起始位，然后按波特率依次发送数据位，最后回到高电平停止位。

UART 这里说的“波特率”通常就是我们最常见的通信速率。面试里如果被问到，常见回答可以是：

  • 常见波特率：9600、19200、38400、57600、115200
  • STM32 里一般通过波特率寄存器分频得到目标速率
  • 双方波特率必须一致，否则接收采样点会逐渐偏移，最后导致乱码

IIC 时序图

IIC 最关键的不是“停止位”，而是开始条件和停止条件。判断边界时一定要看 SCL 为高的那个时刻。

如果再细化一点，IIC 的一字节传输通常可以口述成：START -> 7位地址 + R/W -> ACK -> 8位数据 -> ACK/NACK -> STOP。

IIC 严格说更常用“时钟频率”来描述速率，但面试里很多人也会顺手把它叫成波特率。比较常见的速率有：

  • 标准模式：100 kHz
  • 快速模式：400 kHz
  • Fast Mode Plus：1 MHz
  • High Speed Mode：3.4 MHz

  • 面试里如果继续展开，可以补一句：很多 IIC 设备读寄存器本质上都是“先写寄存器地址，再读数据”

SPI 时序图

SPI 没有像 UART 那样的起始位，也没有像 IIC 那样的开始条件。它更像是：片选拉低后，双方跟着时钟同步移位。

SPI 严格来说通常也不叫“波特率”，而是叫 SCLK 时钟频率。不过面试时如果对方泛泛地问“SPI 波特率”，你也可以这样回答：

  • SPI 速率由主机时钟决定，本质上看的是 SCLK
  • STM32 常通过预分频把外设时钟分成 PCLK/2、/4、/8、/16 这类频率
  • 常见 SPI 速率从几百 kHz 到几 MHz，很多器件能跑到十几 MHz，具体上限要看从设备手册

SPI 常见的收发例子是：STM32 和 SPI Flash 或屏幕控制器通信。主机发命令字节的同时，也能同步收到从机返回的数据。下面也换成标准库风格：

  • 发送：主机发出读取命令
  • 接收：从机在时钟驱动下同步回传数据
  • 标准库里本质也是围绕发送/接收标志位在操作数据寄存器
  • CS 拉高，表示本次传输结束

四、为什么 UART 和 IIC 的边界判断方式不一样

如果面试官继续往下问，UART 通常是最常见的切入点。

UART 属于异步通信，没有单独的时钟线，所以接收方必须自己判断一帧数据什么时候开始、什么时候结束。

于是就有了：

  • 起始位
  • 数据位
  • 可选校验位
  • 停止位

它的典型过程是：

  1. 总线空闲时保持高电平
  2. 发送开始时，先拉低一个起始位
  3. 接着按约定波特率发送数据位
  4. 最后拉高停止位，告诉接收端这一帧结束了

所以起始位和停止位，本质上是给“异步世界”建立边界感。

和 UART 不同，I2C 是同步通信，它有专门的时钟线 SCL。

这时候，接收端不需要自己猜节奏，只要跟着时钟采样就行。

因此 I2C 的重点就不再是“靠停止位补边界”，而是用更明确的总线规则定义：

  • 起始条件：SCL 为高时，SDA 从高变低
  • 停止条件：SCL 为高时，SDA 从低变高

这两个规则之所以重要，是因为它们和普通数据位变化严格区分开了。

也就是说，I2C 并不是随便看一眼高低电平，而是通过“特定时刻的跳变”识别一帧通信的边界。

五、总结

把通信协议单独拎出来之后会发现，很多面试题其实都围绕同一个核心：

  • 你是否知道通信引脚各自负责什么
  • 你是否能讲清楚数据为什么会按这个时序发送
  • 你是否能分清 UART、IIC、SPI 在时钟、边界和速率上的差别
  • 你是否能把库函数背后的寄存器和硬件行为对应起来

如果只是记几个初始化函数，很容易答着答着就卡住。
但如果能从“引脚构成 -> 时序规则 -> 速率 -> 收发过程”这条线往下讲，回答会稳很多。
